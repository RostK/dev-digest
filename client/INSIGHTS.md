## What Works

- 2026-06-16 â€” Lazy "fetch on hover" for PR-list rows: `usePrReviews(prId, { enabled })` gated on a per-row hover flag (set once via the hover card's `onOpen`), sharing the `["reviews", prId]` query key so the count chips show instantly (from `pr.findings`) while the title list loads only when hovered â€” and the PR-detail page is warm afterwards. Prefer adding an optional `{ enabled }` to the existing query hook over writing a second hook. Evidence: client/src/lib/hooks/reviews.ts (usePrReviews), client/src/app/repos/[repoId]/pulls/_components/PRRow/PRRow.tsx.

- 2026-06-17 â€” A portaled `position:fixed` hover card must be placed AFTER it mounts, not from the trigger rect alone: `top = rect.bottom + 6` clips for rows low in the viewport (the exact bug on the last PR-list row). Fix = keep a `cardRef`, and in a `useLayoutEffect` (runs before paint â†’ no position flash) measure BOTH the trigger rect and the rendered card, then (a) flip above the trigger when below would overflow the viewport bottom and there's room up top, (b) clamp `left` so the card stays inside the right edge, (c) keep an 8px margin from all edges. Re-run placement on content-size changes (`findings?.length`/`loading` â€” the lazy fetch grows the card) and on `scroll`/`resize` (listeners added while open). Completes the portal note in "What Doesn't Work" (which fixed `overflow:hidden` clipping but not viewport-edge clipping). Evidence: client/src/components/SeverityIndicators/FindingsHoverCard.tsx (place() + useLayoutEffect).

- 2026-06-17 â€” Click-to-filter on the severity cluster needs PER-ROW stateful UI, but RunHistory renders the cluster inside a `.map()` IIFE where you CAN'T call a hook. Extract a small per-item component (`FindingsCluster`) that owns `useSeverityFilter` + renders `SeverityIndicators` (chips) + `FindingsHoverCard`; the PR list and the timeline both render it, so the interaction is defined once. The hover card supports "hover OR pinned" via `show = open || pinned` (a clicked chip pins it); while pinned, hover-out is a no-op and a document `mousedown` capture listener dismisses it (clicks inside trigger/card are ignored via `ref.contains`). Filtering is client-side (`findings.filter(f => f.severity === active)`); chips toggle (click active again clears). Evidence: client/src/components/SeverityIndicators/{FindingsCluster.tsx, useSeverityFilter.ts, FindingsHoverCard.tsx (pinned)}, RunHistory.tsx / PRRow.tsx (both render FindingsCluster).

## Codebase Patterns

- 2026-06-16 â€” `SeverityBadge` in `compact` mode renders icon + count ONLY (no text label), so it exposes just a bare number to screen readers and is untargetable by `getByText`. Wrap each chip in a span with `title`+`aria-label` (e.g. `"2 Critical"` from `SEV[sev].label`) â€” gives a hover tooltip, an accessible name, AND a stable test handle (`getByTitle("2 Critical")`; `getByLabelText` is unreliable on a plain span). Evidence: client/src/components/SeverityIndicators/SeverityIndicators.tsx.
- 2026-06-16 â€” A PR's "findings" = OPEN findings summed across ALL its reviews, NOT the latest review. A multi-agent review pass writes ONE `reviews` row per agent (General/Security/Performance), each with its own `run_id` and `created_at` seconds apart â€” so "latest single review" silently drops the other agents' findings (caught when a PR's WARNING from one agent vanished because two clean agents reviewed milliseconds later). The PR-detail page is the reference: it does `runs.flatMap(r => r.findings)` over every review. The PR-list cluster (server) and its hover card (`reviewsQ.data.flatMap(r => r.findings)` â†’ `openFindings`) must do the same or list and detail disagree. Evidence: client/src/app/repos/[repoId]/pulls/[number]/page.tsx (allFindings flatMap), PRRow.tsx (hoverFindings), server pulls/routes.ts (severityByPr join).
- 2026-06-17 (CORRECTION to the above) â€” Changed to each agent's LATEST review only, NOT all reviews ("review insights change with code" â€” a re-review supersedes that agent's earlier pass against since-changed code). New shared `latestReviewsPerAgent(reviews)` (in SeverityIndicators/helpers.ts, exported from the barrel) keeps the newest review per `agent_id` (null agent â†’ one bucket, matching the server's `DISTINCT ON (pr_id, agent_id)`); skips `kind!=='review'` so a newer summary can't suppress a review's findings. Apply it at ALL three surfaces so they still agree: PRRow `hoverFindings`, PR-detail `allFindings` (drives findingsCount + lethalTrifecta; the `runs` timeline still shows every run), and the server cluster. The "summed across ALL reviews" wording above is superseded. Evidence: client/src/components/SeverityIndicators/helpers.ts (latestReviewsPerAgent), PRRow.tsx, pulls/[number]/page.tsx (allFindings).

- 2026-06-16 â€” Review-run cards render `ReviewRecord` (reviews table: verdict/score/findings) which has NO tokens or cost â€” those live on `RunSummary` (agent_runs), keyed by `run_id`. PR Detail already fetches both (`usePrReviews` + `usePrRuns`); match `review.run_id â†’ RunSummary` in `FindingsTab` and pass the run into the accordion rather than widening the reviews query. Evidence: client/src/app/repos/[repoId]/pulls/[number]/_components/FindingsTab/FindingsTab.tsx (runById map), ReviewRunAccordion.tsx.
- 2026-06-16 â€” Adding a Pull Requests list column = add the key to `COLUMN_KEYS` + widen the `GRID` template string (both in pulls/constants.ts) + add a cell to PRRow.tsx in the SAME position (cells render in fixed order â€” there is no keyâ†’cell mapping) + add i18n `prReview.list.columns.<key>`. - 2026-06-16 â€” `@devdigest/shared` is a per-app VENDORED copy (client/src/vendor/shared/, via tsconfig path) â€” a zod contract change must be made identically in server/src/vendor/shared/ too; there is no sync script between them. - 2026-06-17 â€” The findings hover card shows "most recent N" by slicing the FRONT of the passed array (`findings.slice(0, max)`, default 5, with a `+N more` footer) â€” there is NO sort inside the card. If you ever change the review ordering, the card's "recent 5" silently degrades to an arbitrary 5 â€” re-sort at the call site, don't assume DB order. Evidence: server/src/modules/reviews/repository/review.repo.ts:66 (desc createdAt), PRRow.tsx (flatMap), components/SeverityIndicators/FindingsHoverCard.tsx (slice + moreLabel).

## Decisions

- 2026-06-16 â€” Per-run cost is formatted at DIFFERENT precision per surface, on purpose: 2â€“3 dp compact (`formatCostCompact`) for the PR-list column + verdict card header, but 4 dp (`formatCostPrecise`) for the dense PR timeline usage line â€” sub-cent runs ($0.0013 vs $0.0014) must stay distinguishable there, where 3 dp would collapse both to "$0.001". Don't "unify" the formatters. Evidence: client/src/components/RunCostBadge/format.ts.
- 2026-06-16 â€” Severity counts are computed in TWO places with DIFFERENT scopes, on purpose: server `rollupSeverities` for the PR-LIST cluster = open findings across ALL of a PR's reviews (per-PR rollup); client `countsOf` for the PR-detail TIMELINE = open findings of ONE run (per-run, keyed by run_id). Both share the OPEN (non-dismissed) rule but are NOT the same aggregation â€” the list answers "what's open on this PR", the timeline row answers "what did this run find". Don't unify them. Evidence: client/src/components/SeverityIndicators/helpers.ts (countsOf), server/src/modules/pulls/status.ts (rollupSeverities), routes.ts (severityByPr).
