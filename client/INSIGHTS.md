## What Works

- 2026-06-16 — Lazy "fetch on hover" for PR-list rows: `usePrReviews(prId, { enabled })` gated on a per-row hover flag (set once via the hover card's `onOpen`), sharing the `["reviews", prId]` query key so the count chips show instantly (from `pr.findings`) while the title list loads only when hovered — and the PR-detail page is warm afterwards. Prefer adding an optional `{ enabled }` to the existing query hook over writing a second hook. Evidence: client/src/lib/hooks/reviews.ts (usePrReviews), client/src/app/repos/[repoId]/pulls/_components/PRRow/PRRow.tsx.

- 2026-06-17 — A portaled `position:fixed` hover card must be placed AFTER it mounts, not from the trigger rect alone: `top = rect.bottom + 6` clips for rows low in the viewport (the exact bug on the last PR-list row). Fix = keep a `cardRef`, and in a `useLayoutEffect` (runs before paint → no position flash) measure BOTH the trigger rect and the rendered card, then (a) flip above the trigger when below would overflow the viewport bottom and there's room up top, (b) clamp `left` so the card stays inside the right edge, (c) keep an 8px margin from all edges. Re-run placement on content-size changes (`findings?.length`/`loading` — the lazy fetch grows the card) and on `scroll`/`resize` (listeners added while open). Completes the portal note in "What Doesn't Work" (which fixed `overflow:hidden` clipping but not viewport-edge clipping). Evidence: client/src/components/SeverityIndicators/FindingsHoverCard.tsx (place() + useLayoutEffect).

- 2026-06-17 — Click-to-filter on the severity cluster needs PER-ROW stateful UI, but RunHistory renders the cluster inside a `.map()` IIFE where you CAN'T call a hook. Extract a small per-item component (`FindingsCluster`) that owns `useSeverityFilter` + renders `SeverityIndicators` (chips) + `FindingsHoverCard`; the PR list and the timeline both render it, so the interaction is defined once. The hover card supports "hover OR pinned" via `show = open || pinned` (a clicked chip pins it); while pinned, hover-out is a no-op and a document `mousedown` capture listener dismisses it (clicks inside trigger/card are ignored via `ref.contains`). Filtering is client-side (`findings.filter(f => f.severity === active)`); chips toggle (click active again clears). Evidence: client/src/components/SeverityIndicators/{FindingsCluster.tsx, useSeverityFilter.ts, FindingsHoverCard.tsx (pinned)}, RunHistory.tsx / PRRow.tsx (both render FindingsCluster).

- 2026-06-24 — "Lazy compute-on-view" (IntentCard): the GET hook is a PURE read whose API returns `Intent | null`, and the card fires a one-shot `recompute` mutation when the value RESOLVES to `null`. Gate on `data === null`, NOT `!data` — `data` is `undefined` while the query is pending, so `!data` would fire the (billable) mutation mid-load and double-call; only `=== null` means "resolved, nothing stored". Guard the single fire with a `useRef` boolean (the mutation object is stable in TanStack Query v5, but the ref is the real once-only guarantee). The same compute service backs the server's `ensureIntent` (get-or-compute before review), so the two paths converge idempotently. Evidence: client/src/app/repos/[repoId]/pulls/[number]/_components/IntentCard/IntentCard.tsx, client/src/lib/hooks/core.ts (usePrIntent/useRecomputeIntent).

- 2026-06-24 — ONE diff renderer serves both "Original" and "Smart order" by adding ADDITIVE OPTIONAL props to the shared primitives instead of forking them: `FileCard` gains `findingLines?`/`defaultOpen?`, `CodeLine` gains `highlight?`/`anchorId?` — absent props = byte-identical original render (the flat `DiffViewer` passes none). Smart Diff finding lines get `id={`sd-${file.path}-L${newNo}`}` and the header "N findings" badge scrolls to the first via `document.getElementById(id)?.scrollIntoView({ block: "center" })` inside `requestAnimationFrame` (so the card is expanded first). Use getElementById, NOT querySelector — the id embeds `/` and `.` from the path, which would need CSS-escaping in a selector. Evidence: client/src/components/diff-viewer/{FileCard/FileCard.tsx, CodeLine/CodeLine.tsx, SmartDiffViewer/SmartDiffViewer.tsx}.

- 2026-06-24 — Smart Diff's per-line severity badges + click-to-expand finding cards are a TOKEN-FREE OVERLAY assembled on the CLIENT from `usePrReviews` — NOT from the smart-diff endpoint. `DiffTab` builds `Map<path, FindingRecord[]>` (via `latestReviewsPerAgent`, non-dismissed) and threads it SmartDiffViewer → FileCard → CodeLine; the smart-diff contract's `finding_lines: number[]` only drives grouping + the line highlight, while the severity AND the finding content (title/rationale/suggestion) come from the reviews overlay. The inline `SeverityBadge` is a real `<button>` (`aria-expanded`) that toggles a local `InlineFinding` card rendered under the line (same slot as comment threads). Keep the overlay LOCAL to the diff-viewer (new `InlineFinding` mirrors the findings-tab `FindingCard`) — do NOT import `app/.../FindingCard` into the shared diff-viewer (shared→page-feature is the wrong dependency direction). Evidence: client/src/app/repos/[repoId]/pulls/[number]/_components/DiffTab/DiffTab.tsx (buildFileFindings), components/diff-viewer/{CodeLine/CodeLine.tsx, InlineFinding/InlineFinding.tsx}.

## What Doesn't Work

- 2026-06-16 — A `position:absolute` dropdown/popover anchored to a PR-list ROW gets clipped: the table card (`s.tableCard`) sets `overflow:hidden` (for rounded row corners), which clips ALL absolutely-positioned descendants. The house `Dropdown` only works there because it sits at the table top and opens downward within the tall card. For a hover card on an arbitrary row, render it via `createPortal(…, document.body)` with `position:fixed` measured from the trigger's `getBoundingClientRect()` (add a small close-delay so crossing the gap to the portaled card doesn't dismiss it). Evidence: client/src/app/repos/[repoId]/pulls/styles.ts (tableCard), client/src/components/SeverityIndicators/FindingsHoverCard.tsx.

- 2026-07-03 — The Claude_Preview MCP and the claude-in-chrome extension can NOT drive the studio for UI verification: both load the app shell but the app's TanStack Query data hooks never fire (ZERO requests to :3001), leaving every route stuck on a loading spinner / "No repo selected". It is NOT connectivity/CORS/auth — a manual `fetch('http://localhost:3001/...')` in the SAME page's console returns 200 (api.ts sends no auth header; the API needs none), and the `firebase:authUser` in localStorage is stale data from a prior app on :3000, unrelated. Use the documented headless-Chrome-over-CDP recipe instead (see Tool & Library Notes 2026-06-17) — it renders the studio fine on the same dev instance and can assert geometry. Cost this session: hours of failed SPEC-04 PrBriefCard verification before falling back to API-level checks. Evidence: this session's SPEC-04 UI-verification attempts; client/INSIGHTS.md Tool & Library Notes (CDP recipe).
  - 2026-07-07 (PARTIAL CONTRADICTION) — claude-in-chrome DID drive the studio this session: the eval dashboards loaded REAL data from :3001 (skeletons → "1 eval case", metric values, the Recharts trend, CompareModal), so its data hooks DO fire. The prior failure was likely transient/environmental (stale :3000 state), not a hard tool limitation. Working recipe: `navigate` then WAIT ~3–5s (queries show skeletons first); prefer `find`(natural language)→ref then `computer{ref}` clicks over screenshot pixel coordinates. Caveats seen: (1) screenshots intermittently `CDP … timed out` — just retry, the 2nd succeeds; (2) the window often renders at 2× DPI so the CSS viewport is small (~760px) and content looks "zoomed", cutting off right-side tabs — `resize_window` helps but is unreliable, so drive by element ref not coordinates; (3) `read_page{filter:"interactive"}` sometimes returns empty mid-load — re-read after the wait or use `find`. Net: usable for real UI verification (Recharts render + modal interaction confirmed), which the 2026-07-03 note said was impossible. Evidence: this session's SPEC-05 Phase-3.5 browser verification.

- 2026-07-06 — Adding a TanStack Query hook (useQuery/useMutation) to a SHARED CHILD component silently breaks every PARENT test that renders it without a `QueryClientProvider` — the child calls `useQueryClient()` UNCONDITIONALLY (hooks run regardless of whether the new button/branch actually renders), so a parent smoke test throws "No QueryClient set". Adding a NEW i18n namespace to that child ALSO breaks parents that don't provide it (`useTranslations("evals")` → MISSING_MESSAGE). Fix the caller test IN THE SAME task unit, mirroring the file's existing idiom: FindingsPanel.test.tsx already `vi.mock`s its data hook, so mock the new hook the same way + add the namespace to the test's `NextIntlClientProvider messages`. A child's OWN test providing inline providers does NOT save its parents. Evidence: FindingCard.tsx (useCreateEvalFromFinding + useTranslations("evals")), FindingsPanel.test.tsx.

## Codebase Patterns

- 2026-06-16 — `SeverityBadge` in `compact` mode renders icon + count ONLY (no text label), so it exposes just a bare number to screen readers and is untargetable by `getByText`. Wrap each chip in a span with `title`+`aria-label` (e.g. `"2 Critical"` from `SEV[sev].label`) — gives a hover tooltip, an accessible name, AND a stable test handle (`getByTitle("2 Critical")`; `getByLabelText` is unreliable on a plain span). Evidence: client/src/components/SeverityIndicators/SeverityIndicators.tsx.
- 2026-06-16 — A PR's "findings" = OPEN findings summed across ALL its reviews, NOT the latest review. A multi-agent review pass writes ONE `reviews` row per agent (General/Security/Performance), each with its own `run_id` and `created_at` seconds apart — so "latest single review" silently drops the other agents' findings (caught when a PR's WARNING from one agent vanished because two clean agents reviewed milliseconds later). The PR-detail page is the reference: it does `runs.flatMap(r => r.findings)` over every review. The PR-list cluster (server) and its hover card (`reviewsQ.data.flatMap(r => r.findings)` → `openFindings`) must do the same or list and detail disagree. Evidence: client/src/app/repos/[repoId]/pulls/[number]/page.tsx (allFindings flatMap), PRRow.tsx (hoverFindings), server pulls/routes.ts (severityByPr join).
- 2026-06-17 (CORRECTION to the above) — Changed to each agent's LATEST review only, NOT all reviews ("review insights change with code" — a re-review supersedes that agent's earlier pass against since-changed code). New shared `latestReviewsPerAgent(reviews)` (in SeverityIndicators/helpers.ts, exported from the barrel) keeps the newest review per `agent_id` (null agent → one bucket, matching the server's `DISTINCT ON (pr_id, agent_id)`); skips `kind!=='review'` so a newer summary can't suppress a review's findings. Apply it at ALL three surfaces so they still agree: PRRow `hoverFindings`, PR-detail `allFindings` (drives findingsCount + lethalTrifecta; the `runs` timeline still shows every run), and the server cluster. The "summed across ALL reviews" wording above is superseded. Evidence: client/src/components/SeverityIndicators/helpers.ts (latestReviewsPerAgent), PRRow.tsx, pulls/[number]/page.tsx (allFindings).

- 2026-06-16 — Review-run cards render `ReviewRecord` (reviews table: verdict/score/findings) which has NO tokens or cost — those live on `RunSummary` (agent_runs), keyed by `run_id`. PR Detail already fetches both (`usePrReviews` + `usePrRuns`); match `review.run_id → RunSummary` in `FindingsTab` and pass the run into the accordion rather than widening the reviews query. Evidence: client/src/app/repos/[repoId]/pulls/[number]/_components/FindingsTab/FindingsTab.tsx (runById map), ReviewRunAccordion.tsx.
- 2026-06-16 — Adding a Pull Requests list column = add the key to `COLUMN_KEYS` + widen the `GRID` template string (both in pulls/constants.ts) + add a cell to PRRow.tsx in the SAME position (cells render in fixed order — there is no key→cell mapping) + add i18n `prReview.list.columns.<key>`. The header row auto-maps labels from COLUMN_KEYS; GRID drives both headRow and rows. Evidence: client/src/app/repos/[repoId]/pulls/constants.ts, _components/PRRow/PRRow.tsx, styles.ts.
- 2026-06-16 — `@devdigest/shared` is a per-app VENDORED copy (client/src/vendor/shared/, via tsconfig path) — a zod contract change must be made identically in server/src/vendor/shared/ too; there is no sync script between them. Evidence: client/tsconfig.json paths; client/src/vendor/shared/contracts/{trace,platform}.ts.

- 2026-06-17 — The findings hover card shows "most recent N" by slicing the FRONT of the passed array (`findings.slice(0, max)`, default 5, with a `+N more` footer) — there is NO sort inside the card. It only means "most recent" because reviews arrive newest-first (server `reviewsForPull` does `orderBy(desc(reviews.createdAt))`) and PRRow flatMaps findings over them in that order. If you ever change the review ordering, the card's "recent 5" silently degrades to an arbitrary 5 — re-sort at the call site, don't assume DB order. The header count (`title`) still reflects the TRUE total, not the capped 5. Evidence: server/src/modules/reviews/repository/review.repo.ts:66 (desc createdAt), PRRow.tsx (flatMap), components/SeverityIndicators/FindingsHoverCard.tsx (slice + moreLabel).

- 2026-06-20 — The sidebar nav is a HARDCODED list in client/src/vendor/ui/nav.ts (the `NAV` array: `{ key, label, icon, href, gKey }`) — NOT i18n-driven and NOT route-derived. `activeKeyFor(pathname)` (components/app-shell/helpers.ts) and `shell.json` `nav.<key>` labels can BOTH already exist for a route that has NO sidebar entry (the starter shipped active-key + labels for skills/conventions/eval/memory but `NAV` only listed pulls+agents). To surface a new top-level route you must add an item to `NAV` (and a `g <key>` line to `SHORTCUTS`), not just `activeKeyFor`. Evidence: client/src/vendor/ui/nav.ts (NAV, SHORTCUTS), components/app-shell/helpers.ts (activeKeyFor).

- 2026-06-20 — i18n namespaces auto-load: i18n/request.ts `loadMessages` `readdirSync`-globs messages/en/*.json into `{ [basename]: json }`, so dropping a new messages/en/<ns>.json makes `useTranslations("<ns>")` work with NO registration step. Caveat: next-intl throws when a `t("…")` key is missing from its namespace, so EVERY key a component references must exist in the json — a partial namespace breaks the whole page, not just one string (cross-check keys ↔ json when adding a feature's messages). Evidence: client/src/i18n/request.ts (loadMessages).

- 2026-06-20 — Creating a Skill FROM another feature (conventions) is done entirely CLIENT-side — there is NO conventions→skills backend endpoint. The view/modal assemble the markdown body + name (lib/convention-skill.ts) from the ACCEPTED candidates and call the existing `useCreateSkill` → `POST /skills` with `source:'extracted'`, `type:'convention'`, `evidence_files`. `POST /skills` already accepted source+evidence_files; the only change needed was adding `evidence_files` to the client hook's `CreateSkillInput`. "Split by category" just loops `createSkill.mutateAsync` once per category. Don't add a server endpoint for this. Evidence: client/src/lib/convention-skill.ts, app/conventions/_components/ConventionsView/_components/CreateSkillFromConventionsModal/CreateSkillFromConventionsModal.tsx, lib/hooks/skills.ts (CreateSkillInput.evidence_files).

Tab nav = `router.replace`, picking a skill = `router.push` preserving the current tab. This SUPERSEDES the 2026-06-20 "skills mirrors agents 1:1" note for the list/detail LAYOUT (agents still keeps grid list + separate /agents/:id page); the modals/hooks/CSS-token styling still mirror agents. A body edit appends a new immutable version, so `useUpdateSkill` must invalidate `["skill-versions", id]` or the Versions tab goes stale. Evidence: client/src/app/skills/page.tsx, [id]/page.tsx, _components/SkillsListView/SkillsListView.tsx + _components/SkillDetail/, lib/hooks/skills.ts (useSkillVersions, useUpdateSkill invalidation).

## Decisions

- 2026-06-16 — Per-run cost is formatted at DIFFERENT precision per surface, on purpose: 2–3 dp compact (`formatCostCompact`) for the PR-list column + verdict card header, but 4 dp (`formatCostPrecise`) for the dense PR timeline usage line — sub-cent runs ($0.0013 vs $0.0014) must stay distinguishable there, where 3 dp would collapse both to "$0.001". Don't "unify" the formatters. Evidence: client/src/components/RunCostBadge/format.ts.
- 2026-06-16 — Severity counts are computed in TWO places with DIFFERENT scopes, on purpose: server `rollupSeverities` for the PR-LIST cluster = open findings across ALL of a PR's reviews (per-PR rollup); client `countsOf` for the PR-detail TIMELINE = open findings of ONE run (per-run, keyed by run_id). Both share the OPEN (non-dismissed) rule but are NOT the same aggregation — the list answers "what's open on this PR", the timeline row answers "what did this run find". Don't unify them. Evidence: client/src/components/SeverityIndicators/helpers.ts (countsOf), server/src/modules/pulls/status.ts (rollupSeverities), routes.ts (severityByPr).

## Recurring Errors & Fixes

- 2026-07-07 — The shared inline `Sparkline` (vendor/ui/charts/Sparkline.tsx) emitted a React `Received NaN for the \`cx\` attribute` warning (and tripped the Next dev error overlay) whenever it got a SINGLE data point: `x = (i / (data.length - 1)) * w` is `0 / 0 = NaN` when `length === 1`. The `y` was already safe (`span = max-min || 1`); only x divided by the point-count. A MetricCard with exactly one eval run hit this. Fix: special-case a lone point to `w / 2` (centered). Any sparkline/line-chart that spreads points over `length - 1` needs a single-point guard. Evidence: client/src/vendor/ui/charts/Sparkline.tsx, Sparkline.test.tsx, commit 451d174.

- 2026-06-24 — A type exported from a component file but imported through that folder's BARREL (`import type { FileFinding } from "../FileCard"`) errors `Namespace '"../FileCard"' has no exported member 'FileFinding'` UNLESS the barrel re-exports it — `FileCard/index.ts` must `export type { FileFinding }`. The barrel forwards the component automatically but NOT sibling type exports. Evidence: client/src/components/diff-viewer/FileCard/index.ts.
