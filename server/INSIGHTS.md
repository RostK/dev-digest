## What Works

- 2026-06-20 â€” Convention extraction GROUNDS every LLM proposal against real code before persisting: `collectSamples` reads the sampled files (root configs + `getConventionSamples` ranked files) into a `pathâ†’content` Map, then a candidate survives ONLY if `normalizeEvidencePath` maps it to a sampled path AND `locateSnippet` finds its snippet (whitespace-normalized; consecutive multi-line, then first-line anchor). This drops both hallucinated RULES and hallucinated PATHS, and the kept line range is taken from the FILE (not the model's claim) so the `file:line` GitHub deep-link is always accurate. Evidence: server/src/modules/conventions/service.ts (collectSamples/extract), helpers.ts (locateSnippet, normalizeEvidencePath).

- 2026-06-27 â€” To review an OPEN PR whose head isn't in the shallow clone, `loadDiff` fetches `pull/<n>/head` via the existing `container.git.fetchPullHead(ref, pull.number)` port and RETRIES `git diff base...headSha` BEFORE falling back to `pr_files`. The fetched head's history connects to the merge-base that IS in the clone (the branch's base commit), so `git diff main...<headSha>` resolves even from a depth-1 clone. Verified on the real clone: `git fetch origin pull/<n>/head:pr-<n>` then `git diff main...<headSha>` â†’ full diff. Evidence: server/src/modules/reviews/diff-loader.ts (loadDiff fetch-and-retry), adapters/git/simple-git.ts:72 (fetchPullHead).

- 2026-07-03 â€” Atomic "never clobber a good row with a degraded one" WITHOUT a read-then-write TOCTOU: the degraded/fallback path persists via `INSERT â€¦ ON CONFLICT (pr_id) DO NOTHING â€¦ RETURNING` (a new `insertBriefIfAbsent` whose `.returning()` reveals whether a row was written), while the good path keeps the normal `onConflictDoUpdate` upsert. A prior read-existing-then-conditionally-upsert let two concurrent POSTs (one degraded) overwrite a just-written good brief; the insert-if-absent closes the window at the DB. First `onConflictDoNothing().returning()` combo in the codebase. Evidence: server/src/modules/brief/repository.ts (insertBriefIfAbsent, upsertBrief), service.ts (degraded â†’ insertIfAbsent, good â†’ upsert).

## What Doesn't Work

- 2026-06-16 â€” POSIX-only path/URL string assumptions keep breaking on Windows (dev AND CI run on Windows here). Seen twice: building a file URL via `` `file://${process.argv[1]}` `` (migrate.ts) and deriving a parent dir via `lastIndexOf('/')` (indexer test). Always use node:url `pathToFileURL`/`fileURLToPath` for file URLs and node:path `dirname`/`join` for paths â€” never hand-concatenate `/` or `file://`. Evidence: server/src/db/migrate.ts; test/indexer-pipeline.test.ts.

- 2026-06-27 â€” The review pipeline SILENTLY produced a confident `approve`/score 100 on an EMPTY diff. When `loadDiff` can't obtain the diff it returns an empty `UnifiedDiff`, and reviewer-core's `reviewPullRequest` has NO empty-diff short-circuit â€” single-pass sends the empty diff to the LLM, which "approves" nothing it never saw. An OPEN PR hit this because `loadDiff` sourced the diff from `git diff base...headSha` on the LOCAL clone (shallow, default-branch-only: `sync()` advances only `origin/<default>`, nothing fetched the PR head) AND the `pr_files` fallback was empty until the detail page was opened (see Codebase Patterns). Diagnostic: a grounded review of a large diff burns tens of thousands of tokens; a ~5k-token "approve" means the diff never reached the model. Fixed â€” fetch the head + retry (What Works), and fail the run on a zero-file diff (Decisions). Evidence: reviewer-core/src/review/run.ts:154-180, server/src/modules/reviews/diff-loader.ts.

- 2026-07-03 â€” `wrapUntrusted(label, content)` (@devdigest/reviewer-core) fences the CONTENT but does NOT escape the LABEL â€” it emits `<untrusted source="${label}">â€¦`. Interpolating attacker-influenced data into the label (e.g. a spec-doc PATH from the PR body â†’ `wrapUntrusted(\`spec:${doc.path}\`, â€¦)`) lets a crafted value containing `"`/`>` close the tag and break text out of the sandbox â€” the exact injection the fence exists to stop. ALWAYS pass a CONSTANT label ('linked-issue', 'spec'); put untrusted values INSIDE the content. Same trap for the linked-issue TITLE: fence title+body TOGETHER under one constant-label block, never emit the title in a plain `## Linked issue #N: {title}` header outside the fence. Evidence: server/src/modules/brief/helpers.ts (buildBriefMessages â€” constant labels, title+body co-fenced).

- 2026-07-06 â€” A raw NUL byte written into a `.ts` SOURCE file makes git treat the WHOLE file as BINARY (no text diff, fragile committed blob), even though `tsc`/`vitest` tolerate it so tests stay green and you never notice. An eval-scorer dedup key used a raw NUL as its field delimiter (``` `${f.file}<NUL>${f.start}<NUL>${f.end}` ```): `file` reported "data", `git diff` showed "Bin 7744 -> â€¦". Never embed a raw control char in source â€” use a visible delimiter (a SPACE is collision-safe when the other fields are spaceless integers, since the two rightmost tokens uniquely recover them) or a `\\u0000` escape. Gotcha: an editor / JSON tool layer may re-decode a literal `\\u0000` back into an actual NUL, so normalize such a file in the SHELL (`tr '\\000' ' '`), not an editor. Evidence: server/src/modules/evals/scoring.ts:66 (dedupeExpected), commit e7bce14.

- 2026-07-07 â€” Storing a per-row RATIO and re-aggregating it with `avg()` at the group level SILENTLY violates a "pooled" metric spec: a mean-of-ratios â‰  the pooled ratio `sum(num)/sum(den)` whenever rows have unequal denominators. SPEC-05 AC-7 requires `citation_accuracy = sum(kept)/(sum(kept)+sum(dropped))` pooled; the pure `scoring.ts aggregateRun` did this right, but each `eval_runs` row persisted only the per-case ratio and `repository.getGroupAggregate`/`agentSummaryRows` served `avg(citation_accuracy)` â€” so every dashboard/compare/global number was avg-of-ratios, and the counts to re-pool were already lost. This passed a `/code-review`, TWO plan-verifier passes, AND a live runtime check because every seeded/real row had `dropped=0` (so avg==pooled==1 trivially) â€” an aggregation bug is invisible until a test case has UNEQUAL denominators. Fix: persist the raw counts (`kept`/`dropped` int cols, migration 0019) and pool in SQL `case when sum(kept)+sum(dropped)=0 then 1 else sum(kept)::float8/(sum(kept)+sum(dropped)) end`; recall/precision correctly STAY `avg` (AC-5/AC-6 are means). Whenever a spec says "pooled"/"weighted", store the numerator+denominator, never just the ratio, and add a test with mixed denominators. Evidence: server/src/modules/evals/repository.ts (getGroupAggregate/agentSummaryRows), db/schema/eval.ts (kept/dropped), evals.it.test.ts (AC-7 pooled 0.4â‰ avg 0.625), commit 12eafe0.

- 2026-07-07 â€” A route querystring/param validated as `z.string().min(1)` that maps to a `uuid` COLUMN returns a 500, not a clean 4xx, on a malformed value: the non-uuid string passes zod, reaches the Drizzle query, and Postgres raises `invalid input syntax for type uuid` (an UNCAUGHT PostgresError â†’ opaque 500). `GET /evals/compare?a=&b=` did exactly this â€” the route comment even justified `.min(1)` as "avoid over-constraining a value-object id", but the ids are `randomUUID()`s stored in `eval_runs.group_id uuid`, i.e. always UUIDs. Mocks/unit tests hid this â€” only a live HTTP call with a non-uuid surfaced it. Evidence: server/src/modules/evals/routes.ts (CompareQuery), commit 40c7445.

- 2026-07-07 â€” `waitForPrRuns` (test/helpers/runs.ts) SILENTLY returned the still-running rows on timeout (default 10s), so a slow background review surfaced downstream as a confusing DATA bug, not a timeout: the test read the run's trace before it was persisted â†’ `GET /runs/:id/trace` 404 â†’ `trace.specs_read` (etc.) came back `undefined` and the assertion read `expected undefined to equal [...]`. - 2026-07-09 â€” A "concurrent" multi-agent fan-out (N separate `container.reviewService.runReview` calls under `Promise.all`, each firing its OWN fire-and-forget `executeRuns`) does NOT run concurrently on a COLD PR â€” it SERIALIZES on the shared intent pre-work. - 2026-07-09 â€” A free-form `z.array(z.string())` contract field that gets INTERPOLATED into generated YAML is a YAML-injection hole. `CiExportInput.triggers` fed straight into the workflow template `on: pull_request: types: [${triggers.join(', ')}]` let a crafted value break out of the array and inject top-level keys (`pull_request_target`, a comment trigger) into the emitted `.github/workflows/devdigest-review.yml` â€” defeating the exact AC-6/7/8 security invariants the generator enforces.
