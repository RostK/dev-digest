## What Works

- 2026-06-20 — Convention extraction GROUNDS every LLM proposal against real code before persisting: `collectSamples` reads the sampled files (root configs + `getConventionSamples` ranked files) into a `path→content` Map, then a candidate survives ONLY if `normalizeEvidencePath` maps it to a sampled path AND `locateSnippet` finds its snippet (whitespace-normalized; consecutive multi-line, then first-line anchor). This drops both hallucinated RULES and hallucinated PATHS, and the kept line range is taken from the FILE (not the model's claim) so the `file:line` GitHub deep-link is always accurate. Evidence: server/src/modules/conventions/service.ts (collectSamples/extract), helpers.ts (locateSnippet, normalizeEvidencePath).

- 2026-06-27 — To review an OPEN PR whose head isn't in the shallow clone, `loadDiff` fetches `pull/<n>/head` via the existing `container.git.fetchPullHead(ref, pull.number)` port and RETRIES `git diff base...headSha` BEFORE falling back to `pr_files`. The fetched head's history connects to the merge-base that IS in the clone (the branch's base commit), so `git diff main...<headSha>` resolves even from a depth-1 clone. Verified on the real clone: `git fetch origin pull/<n>/head:pr-<n>` then `git diff main...<headSha>` → full diff. Evidence: server/src/modules/reviews/diff-loader.ts (loadDiff fetch-and-retry), adapters/git/simple-git.ts:72 (fetchPullHead).

- 2026-07-03 — Atomic "never clobber a good row with a degraded one" WITHOUT a read-then-write TOCTOU: the degraded/fallback path persists via `INSERT … ON CONFLICT (pr_id) DO NOTHING … RETURNING` (a new `insertBriefIfAbsent` whose `.returning()` reveals whether a row was written), while the good path keeps the normal `onConflictDoUpdate` upsert. A prior read-existing-then-conditionally-upsert let two concurrent POSTs (one degraded) overwrite a just-written good brief; the insert-if-absent closes the window at the DB. First `onConflictDoNothing().returning()` combo in the codebase. Evidence: server/src/modules/brief/repository.ts (insertBriefIfAbsent, upsertBrief), service.ts (degraded → insertIfAbsent, good → upsert).

## What Doesn't Work

- 2026-06-16 — POSIX-only path/URL string assumptions keep breaking on Windows (dev AND CI run on Windows here). Seen twice: building a file URL via `` `file://${process.argv[1]}` `` (migrate.ts) and deriving a parent dir via `lastIndexOf('/')` (indexer test). Always use node:url `pathToFileURL`/`fileURLToPath` for file URLs and node:path `dirname`/`join` for paths — never hand-concatenate `/` or `file://`. Evidence: server/src/db/migrate.ts; test/indexer-pipeline.test.ts.

- 2026-06-27 — The review pipeline SILENTLY produced a confident `approve`/score 100 on an EMPTY diff. When `loadDiff` can't obtain the diff it returns an empty `UnifiedDiff`, and reviewer-core's `reviewPullRequest` has NO empty-diff short-circuit — single-pass sends the empty diff to the LLM, which "approves" nothing it never saw. An OPEN PR hit this because `loadDiff` sourced the diff from `git diff base...headSha` on the LOCAL clone (shallow, default-branch-only: `sync()` advances only `origin/<default>`, nothing fetched the PR head) AND the `pr_files` fallback was empty until the detail page was opened (see Codebase Patterns). Diagnostic: a grounded review of a large diff burns tens of thousands of tokens; a ~5k-token "approve" means the diff never reached the model. Fixed — fetch the head + retry (What Works), and fail the run on a zero-file diff (Decisions). Evidence: reviewer-core/src/review/run.ts:154-180, server/src/modules/reviews/diff-loader.ts.

- 2026-07-03 — `wrapUntrusted(label, content)` (@devdigest/reviewer-core) fences the CONTENT but does NOT escape the LABEL — it emits `<untrusted source="${label}">…`. Interpolating attacker-influenced data into the label (e.g. a spec-doc PATH from the PR body → `wrapUntrusted(\`spec:${doc.path}\`, …)`) lets a crafted value containing `"`/`>` close the tag and break text out of the sandbox — the exact injection the fence exists to stop. ALWAYS pass a CONSTANT label ('linked-issue', 'spec'); put untrusted values INSIDE the content. Same trap for the linked-issue TITLE: fence title+body TOGETHER under one constant-label block, never emit the title in a plain `## Linked issue #N: {title}` header outside the fence. Evidence: server/src/modules/brief/helpers.ts (buildBriefMessages — constant labels, title+body co-fenced).

- 2026-07-06 — A raw NUL byte written into a `.ts` SOURCE file makes git treat the WHOLE file as BINARY (no text diff, fragile committed blob), even though `tsc`/`vitest` tolerate it so tests stay green and you never notice. An eval-scorer dedup key used a raw NUL as its field delimiter (``` `${f.file}<NUL>${f.start}<NUL>${f.end}` ```): `file` reported "data", `git diff` showed "Bin 7744 -> …". Never embed a raw control char in source — use a visible delimiter (a SPACE is collision-safe when the other fields are spaceless integers, since the two rightmost tokens uniquely recover them) or a `\\u0000` escape. Gotcha: an editor / JSON tool layer may re-decode a literal `\\u0000` back into an actual NUL, so normalize such a file in the SHELL (`tr '\\000' ' '`), not an editor. Evidence: server/src/modules/evals/scoring.ts:66 (dedupeExpected), commit e7bce14.

- 2026-07-07 — A route querystring/param validated as `z.string().min(1)` that maps to a `uuid` COLUMN returns a 500, not a clean 4xx, on a malformed value: the non-uuid string passes zod, reaches the Drizzle query, and Postgres raises `invalid input syntax for type uuid` (an UNCAUGHT PostgresError → opaque 500). `GET /evals/compare?a=&b=` did exactly this — the route comment even justified `.min(1)` as "avoid over-constraining a value-object id", but the ids are `randomUUID()`s stored in `eval_runs.group_id uuid`, i.e. always UUIDs. Validate any id that backs a uuid column as `.uuid()` so a malformed value is 422'd at the boundary (a valid-but-unknown id still 404s in the service). Mocks/unit tests hid this — only a live HTTP call with a non-uuid surfaced it. Evidence: server/src/modules/evals/routes.ts (CompareQuery), commit 40c7445.

## Codebase Patterns

- 2026-06-16 — `@devdigest/shared` is vendored INDEPENDENTLY into server/src/vendor/shared/ and client/src/vendor/shared/ (one tsconfig path per app, NO sync script) — a contract change (new zod field, etc.) must be applied to BOTH copies or the apps drift. Ignore server/clones/RostK/dev-digest/** (a nested self-review checkout, not part of the build). Evidence: server/tsconfig.json paths; server/src/vendor/shared/contracts/{trace,platform}.ts.
- 2026-06-16 — GET routes declare only `params`/`body` schemas (fastify-type-provider-zod validates INPUT, not output), so a handler's returned object is serialized as-is — surface a new computed field (e.g. cost_usd) by adding it to the returned object + the vendored zod type, no response-schema change needed. Trade-off: response/contract drift is NOT caught at the route boundary; keep the vendored contract in sync by hand. Evidence: server/src/modules/pulls/routes.ts, server/src/modules/reviews/routes.ts.
- 2026-06-16 — For "latest row per PR" lookups use `db.selectDistinctOn([t.agentRuns.prId], {…}).where(… status='done').orderBy(t.agentRuns.prId, desc(t.agentRuns.ranAt))` — one row per PR resolved in Postgres — NOT fetch-all-rows-then-dedup-in-JS (unbounded as re-reviews accumulate). agent_runs had NO prId index; added composite (pr_id, status, ran_at) where the status equality also satisfies the (pr_id, ran_at) ordering. NB the latest-review-score lookup in the same route still uses the JS-dedup pattern (pre-existing). Evidence: server/src/modules/pulls/routes.ts (latestRunCostByPr), server/src/db/schema/runs.ts (agent_runs_pr_status_ran_at_idx).
- 2026-06-16 — The pulls-list per-PR FINDINGS severity breakdown counts OPEN findings across ALL of a PR's reviews, NOT the latest one: `db.select({prId: reviews.prId, severity: findings.severity}).from(findings).innerJoin(reviews, eq(findings.reviewId, reviews.id)).where(and(inArray(reviews.prId, prIds), isNull(findings.dismissedAt)))`, then bucket by prId + `rollupSeverities`. A multi-agent pass writes ONE `reviews` row per agent, so scoping to the latest single review drops the other agents' findings AND disagrees with the PR-detail page (which flatMaps findings over every review). Surfaced via the pre-existing `PrMeta.findings` contract field (GET routes serialize the returned object; no response schema). Evidence: server/src/modules/pulls/routes.ts (severityByPr), server/src/modules/pulls/status.ts (rollupSeverities).
- 2026-06-17 (CORRECTION to the above + the `findings: null` Decision below) — The rollup NO LONGER counts across ALL reviews. It now counts open findings from each agent's LATEST review only: a `DISTINCT ON (pr_id, agent_id) … ORDER BY pr_id, agent_id, created_at DESC` subquery (kind='review') picks the winning reviews, then findings JOIN that + GROUP BY pr_id, severity. Reason: "review insights change with code" — a re-review supersedes that agent's earlier pass, so findings against since-changed code stop counting. Still group by AGENT (not a single latest review, which drops sibling agents). Postgres treats NULL agent_id as one bucket in DISTINCT ON; the client mirror (`latestReviewsPerAgent`) does the same, so the list cluster, its hover card, and the PR-detail aggregate all agree. Verified on real multi-pass data (PR #5 went 9+ all-review findings → 0 once its latest passes were clean). Evidence: server/src/modules/pulls/routes.ts (severityByPr latestPerAgent), client/src/components/SeverityIndicators/helpers.ts (latestReviewsPerAgent).
- 2026-06-17 — Postgres does NOT auto-index a foreign key's REFERENCING column (only the referenced PK). Any FK column you JOIN or filter on in a hot path needs an explicit index in the Drizzle schema (3rd `pgTable` arg → `index('name').on(...)`), then `pnpm db:generate`. Seen twice now: `agent_runs.pr_id` (cost query) and `findings.review_id` (pulls-list severity rollup → `findings_review_dismissed_idx` on `(review_id, dismissed_at)`, migration 0011). Lead the composite with the join key, then the filter column (`dismissed_at IS NULL`); on tiny seed tables the planner still seq-scans (verify with `SET enable_seqscan=off`), the index kicks in as the table grows. Evidence: server/src/db/schema/reviews.ts (findings index), runs.ts (agent_runs index), modules/pulls/routes.ts (severityByPr).

- 2026-06-20 — Skills reach the review prompt via ONE path: `AgentsRepository.enabledSkillBodies(agentId)` (joins agent_skills × skills WHERE `agent_skills.enabled AND skills.enabled`, `orderBy agent_skills.order`) → `run-executor.runOneAgent` passes them as `reviewPullRequest({ skills })` → reviewer-core's `assemblePrompt` ALREADY renders the `## Skills / rules` block AND records `assembly.skills`, which run-executor feeds verbatim into the trace's `prompt_assembly`. So surfacing a prompt slot (skills/memory/specs) is a SERVER-only change — resolve the strings + pass them in; the engine side is pre-wired, no reviewer-core edit. A disabled binding OR a globally-disabled skill is just omitted (why the trace block shows for enabled-only). Evidence: server/src/modules/agents/repository.ts (enabledSkillBodies), modules/reviews/run-executor.ts (runOneAgent skills), reviewer-core/src/review/run.ts:132 + prompt.ts:88,109,131.

- 2026-06-24 (EXCEPTION to the above "engine side is pre-wired" note) — That holds for skills/memory/specs/callers/repoMap but NOT for the intent slot: `PromptParts`/`ReviewInput` had no `intent` field, so the Intent Layer needed reviewer-core edits in ADDITION to the server wiring. Rule: grep `reviewer-core/src/prompt.ts` `PromptParts` first — if your slot isn't there, it's a two-package change. Evidence: reviewer-core/src/prompt.ts (PromptParts.intent added this session).

- 2026-06-24 — The course starter PRE-SHIPS the entire data layer of an upcoming lesson; the lesson is a WIRING job, not a from-scratch build. Confirmed 3× now (skills, conventions, intent): for the Intent Layer the `pr_intent` table (db/schema/reviews.ts), the `Intent`/`PrIntentRecord` zod contracts (vendor/shared/contracts/{brief,review-api}.ts), the `getIntent`/`upsertIntent` repo methods (modules/reviews/repository/pull.repo.ts:49-68), the `FEATURE_MODELS.review_intent` registry entry, AND the Settings model-picker UI all existed empty/unused. ALWAYS search the schema + contracts + repo for an existing table/type/method before designing a new one — "empty tables / unused slots are intentional" (root CLAUDE.md). Evidence: server/src/db/schema/reviews.ts (pr_intent), modules/reviews/repository/pull.repo.ts (getIntent/upsertIntent), vendor/shared/contracts/platform.ts (review_intent).

- 2026-06-20 — To read a repo's identity (owner/name/clonePath/defaultBranch) from a NON-repos module, do a workspace-scoped `db.select().from(t.repos).where(and(eq(workspaceId), eq(id)))` inside your OWN repository — that's the established pattern (pulls/reviews/polling/workspace all do it), NOT importing repos' `RepoRepository` (server/CLAUDE.md forbids importing a sibling module's internals). The "ONLY place that touches the repos table" comment in repos/repository.ts is aspirational, not enforced — a repository owning its SQL over a shared table is fine. Evidence: server/src/modules/conventions/repository.ts (getRepo), modules/reviews/repository/pull.repo.ts:25, modules/pulls/routes.ts.

- 2026-06-27 — A PR's diff has TWO independent sources that can DISAGREE: (1) the REVIEW path computes `git diff base...headSha` from the local git clone (`loadDiff` → `container.git.diff`), and (2) the UI "Files changed"/Intent path uses the `pr_files` patches imported from the GitHub API — and `pr_files` is written ONLY by `GET /pulls/:id` (the detail handler, behind `if (detail.files.length>0)`), NOT by the PR-list import or the review trigger. So the studio can render a full diff (the detail route also returns LIVE GitHub detail) while a review of the SAME open PR sees nothing (clone lacks the head, pr_files not yet written). NB one `pr_files` row can have a NULL patch (GitHub omits the patch for large files, e.g. lock files); `diffFromPrFiles` skips those. Evidence: server/src/modules/pulls/routes.ts:251-262 (pr_files write), modules/reviews/diff-loader.ts (loadDiff/diffFromPrFiles).

- 2026-06-27 — Blast Radius (L04) is a 4th confirmation of the pre-ship pattern: `repoIntel.getBlastRadius(repoId, changedFiles)` (repo-intel/service.ts:220) ALREADY computes changed symbols + callers (capped 20/symbol via MAX_CALLERS_PER_SYMBOL, rank-sorted, decl file excluded) + endpoints/crons, with a persistent-vs-ripgrep degraded fallback — so the new `blast/` module is pure WIRING: route → `pr_files` paths → facade → mapper → summary, NO algorithm. The catch is a SHAPE MISMATCH: the facade's `BlastResult` is flat camelCase (`callers[]` each tagged with the changed symbol it reaches via `viaSymbol`; flat `impactedEndpoints[]`), while the `BlastRadius` contract is nested snake_case (`downstream[]` per symbol). `mapToBlastRadius` groups callers by `viaSymbol` and attributes endpoints/crons per symbol from `factsByFile` (persistent path) OR the flat `impactedEndpoints` (degraded path, where per-file facts are absent). Evidence: server/src/modules/blast/{service.ts,mapper.ts}, repo-intel/service.ts:220, repo-intel/types.ts:74 (BlastResult).

- 2026-06-29 — When the repo index has NO resolved cross-file references (`decl_file` resolution + `file_edges` are later-lesson slots → `references.decl_file` all NULL), the facade's persistent `getBlastRadius` returns symbols but ZERO callers and does NOT fall back to ripgrep while status is `full`. The blast MODULE bridges this with a best-effort resolver (pure reads, no index mutation), triggered only when `result.callers.length === 0`: `getUnambiguousNames` (names declared in exactly ONE file → avoids same-name false positives) → `getCallersByName` (references TO those names, capped PER SYMBOL at the DB via a `row_number() over (partition by to_symbol order by coalesce(file_rank.rank,0) desc)` window — NULLS-LAST + bounded, since on this path most caller files have no rank row) → `buildFallbackResult` folds them into a `BlastResult` for the existing mapper. Evidence: server/src/modules/blast/{service.ts (build fallback branch), repository.ts (getCallersByName window), fallback.ts}.

- 2026-06-29 — Extending the shared `SmartDiffRole` enum is an ATOMIC 5-place change or it breaks: (1+2) the enum in BOTH vendor/shared/contracts/brief.ts copies; (3) `classifyFile` priority + `byRole` init + `ROLE_ORDER` in modules/reviews/smart-diff.ts; (4) the pattern arrays in smart-diff-constants.ts; (5) the client SmartDiffViewer's THREE `Record<SmartDiffRole,…>` maps (exhaustive → a missing role is a TS error) + the matching `smartDiff.*Label/*Desc` keys in messages/en/prReview.json (next-intl throws at runtime on a missing key). Use `as const satisfies Record<SmartDiffRole,string>` on the viewer maps to keep exhaustiveness AND literal-key checking. Did 3→5 (added config, test). Evidence: contracts/brief.ts ×2, modules/reviews/smart-diff{.ts,-constants.ts}, client SmartDiffViewer.tsx, messages/en/prReview.json.

- 2026-07-02 — Project Context (SPEC-02) is a 5th confirmation of the 2026-06-20 "specs prompt slot is server-only, engine pre-wired" rule: `PromptParts.specs?: string[]` stays a FLAT list and reviewer-core is UNTOUCHED. The own-vs-inherited two-block grouping is done SERVER-SIDE in a new `ProjectContextService` (modules/reviews/project-context.ts) that reads each doc via `container.git.readFile` + counts tokens via `container.tokenizer`, emitting ≤2 flat entries (own first, then inherited) with the sub-block label rendered INSIDE each entry so it lands inside the engine's `wrapUntrusted` fence — untrusted doc text therefore can't forge a sub-block label. Empty effective set → `specs: []` → run-executor omits the key → prompt byte-identical (AC-11). Constructed + injected into `ReviewRunExecutor` exactly like `IntentService`. Evidence: server/src/modules/reviews/project-context.ts, run-executor.ts (omit-when-empty + specs_read/specs_tokens), service.ts.

- 2026-07-02 — modules/reviews now has TWO `isSafeRepoPath` path-traversal guards with DIFFERENT semantics, both guarding `container.git.readFile` reads from the same clone: reviews/helpers.ts (NEW — `.md`-only + a `resolve()`-prefix inside-clone check, for Project Context docs) vs intent-service.ts:161 (private — md/mdx/txt/rst, NO resolved-path check, for PR-body spec refs). Kept separate on purpose (different extension policy), but the divergence is a trap: hardening one does NOT harden the other — if you touch either guard, check both. - 2026-07-03 (update to the above) — SPEC-04 T2 LIFTED the intent-service copy of `isSafeRepoPath` + the PR-body ref loaders (`loadLinkedIssue`/`loadSpecDocs`, `SPEC_EXTS`, char caps) verbatim to `server/src/modules/_shared/pr-body-refs.ts` — free functions taking `container`/ports as PARAMS (the sanctioned cross-module reuse location, per `_shared/project-context.ts`), now shared by BOTH intent-service AND the brief module. So the intent/brief guard now lives in `_shared`; the reviews/helpers.ts guard (md-only + resolved-path check) is STILL a separate divergent copy. - 2026-07-03 — The flip side of the "GET serializes the returned object, no response schema" pattern above: a repository READ of an unconstrained `jsonb` column that flows out such a GET must `Contract.safeParse(row.json)` and return null on failure, NOT `row.json as Contract`. The write path is contract-shaped, but a row from an older lesson/shape, a manual seed, or a future contract change would otherwise serialize OFF-CONTRACT straight to the client (nothing downstream re-checks). safeParse-at-read is the only boundary backstop. Evidence: server/src/modules/brief/repository.ts (getBrief → Brief.safeParse).

- 2026-07-03 — To let a SIBLING module read a PR's blast MAP with ZERO LLM calls, expose a summary-free variant on the container facade: `container.blast.blastMapForPr(workspaceId, prId)` runs the same `build` pipeline (incl. the name-match caller fallback) but with `deterministicSummary` instead of the model `summarize`. Register via a lazy `get blast()` getter + a `blast?` `ContainerOverrides` entry for test injection (mirrors `container.repoIntel`); refactor `build` to take a `SummaryStrategy` so both variants share it. Evidence: server/src/platform/container.ts (get blast), modules/blast/service.ts (blastMapForPr, build(strategy)).

## Decisions

- 2026-06-20 — Skill import is parsed in-memory, body-only, and NEVER runs the archive: the route takes JSON `{ filename, content_base64 }` (NOT @fastify/multipart — one less dep/config) with a raised `bodyLimit: 2_000_000` to fit the base64 envelope, while the service caps the DECODED bytes (`MAX_IMPORT_BYTES` ~1 MB); `fflate.unzipSync` runs with a `filter` that returns FALSE for non-markdown entries, so they are listed (preview `ignored_files`) but never decompressed, written, or executed. Save is a separate `POST /skills` after the user confirms the preview (imports default `enabled=false`, `source='community'`). Evidence: server/src/modules/skills/{routes.ts (import/preview bodyLimit), helpers.ts (parseSkillImport filter), service.ts (importPreview cap)}.

- 2026-06-16 — Per-run cost is DERIVED at read-time (`tokens × pricing` via `runCostUsd`), never stored on `agent_runs` — its cost column was dropped (migration 0009 / commit `d45ab0d` "keep model pricing"). Compute it in the GET-route mappers: `listRunsForPull`, the pulls-list route (PR's latest `done` run), and `getRunTrace` stats enrichment. `ci_runs` is the exception — it keeps a real `cost_usd` column (external CI reports cost). Failed/unpriced/0-token runs → null → "—", never "$0.00". Evidence: server/src/adapters/llm/pricing.ts (runCostUsd), modules/reviews/repository/run.repo.ts, modules/pulls/routes.ts, modules/reviews/service.ts.
- 2026-06-16 — Settings test-connection validates an OpenRouter key via the
  AUTHENTICATED `GET /api/v1/key` (and rejects Provisioning keys), NOT `listModels()`.
  Reason: OpenRouter's `/models` is public, so the old `listModels()` check showed a
  green "OK — N models" for revoked/wrong keys that then 401 on every review. Evidence:
  server/src/modules/settings/routes.ts (openrouter branch).
- 2026-06-16 — A PR with zero OPEN findings across ALL its reviews returns `findings: null` on the list — same as never-reviewed — because the severity bucket map simply has no entry for it. Accepted: the client cluster renders "—" for both null and all-zero, so they're visually identical; not worth a "reviewed, 0 findings" sentinel. NB this means score and findings can disagree (a PR can read score 100 from its latest clean agent yet show a WARNING that a sibling agent in the same pass flagged) — that's correct, and matches the detail page. Evidence: server/src/modules/pulls/routes.ts (`severityByPr.get(r.id) ?? null`).

- 2026-06-20 — A system-LLM feature picks its model via ONE of two settings helpers, used for different intents: `resolveFeatureModel(id)` = workspace override ELSE the static registry default (`FEATURE_MODELS`); `getFeatureModelOverride(id)` = override-or-undefined, for a caller that keeps its OWN default. The conventions extractor uses the latter `?? CONVENTIONS_DEFAULT_MODEL` (openai/gpt-4.1) so extraction stays CHEAP regardless of the registry — and I lowered the registry default gpt-5.4→gpt-4.1 in ALL THREE copies (server vendor/shared/contracts/platform.ts + client vendor copy + client lib/feature-models.ts mirror) for the Settings UI. feature-models.ts's own doc anticipates conventions as the dynamic-default caller. Evidence: server/src/modules/settings/feature-models.ts, modules/conventions/{service.ts,constants.ts}.

- 2026-06-24 — Smart Diff (L03) is a PURE, zero-token read: `GET /pulls/:id/smart-diff` → `ReviewService.smartDiff` reuses `getPrFiles` + `reviewsForPull` (newest review per `agent_id`, non-dismissed findings — the SAME rollup as the pulls list) and composes them with the pure `composeSmartDiff`; it NEVER calls `container.llm` (the expensive pass already happened in the Structured Reviewer). The file-role classifier (`classifyFile`) + its patterns/thresholds live in the reviews MODULE (`smart-diff.ts` / `smart-diff-constants.ts`), NOT reviewer-core — reviewer-core is the LLM engine, and a deterministic path-based classifier with no model call belongs beside its only consumer (mirrors the Intent Layer's module placement). Evidence: server/src/modules/reviews/{smart-diff.ts, smart-diff-constants.ts, service.ts (smartDiff), routes.ts}.

- 2026-06-24 — Smart Diff fills `SmartDiffFile.pseudocode_summary` (the "What this does" line) TOKEN-FREE by REUSING the Intent Layer's already-computed output: `composeSmartDiff(files, findings, intent)` runs a pure `summarizeFile(path, intent)` that matches each file to its best-fit `intent.in_scope` AREA by token overlap (path segments + filename stem vs area words; prefix-match so webhook≈webhooks, config≈configuration); the service reads the STORED intent via `this.repo.getIntent(prId)` and NEVER computes it (computing is the Intent feature's own pass). So per-file "purpose" comes from intent and per-line "severity" from review findings — both already paid for; the smart-diff path stays zero-model-call. Caveat: `in_scope` is per-AREA, so the match is coarse (a generic area like "Public API endpoints" can match several files). Evidence: server/src/modules/reviews/smart-diff.ts (summarizeFile), service.ts (smartDiff getIntent).

- 2026-06-27 — An empty diff FAILS the run, it is NEVER reviewed: `run-executor` guards `diff.files.length === 0` right after the "Diff ready" log and calls the existing `failAll(...)` (marks every queued run failed + persists the trace). Rationale: reviewing nothing yields a meaningless but CONFIDENT "approve" that masks a data problem (head not fetched / patches missing) as a clean bill of health — failing loudly with a clear message is correct, and the guard backstops any residual empty result from `loadDiff`.
